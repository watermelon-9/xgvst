## 2) 差异清单与修复建议

### 差异 A：`svelte.config.js` 未显式声明 Cloudflare routes 策略
- **现状**：`adapter()` 无参数。
- **风险**：与主线注意事项不一致；后续新增动态 API 路由时，可能出现 Pages/Functions 路由归类不清导致 404。
- **建议修复**：在 `apps/web/svelte.config.js` 增加显式策略（示例）：

```js
adapter({
  routes: {
    include: ['/*'],
    exclude: ['<all>']
  }
})
```

> 实际 include/exclude 需按最终路由策略确认；若当前阶段仅前端页面，也建议先写明策略以防后续冲突。

### 差异 B：`vite.config.ts` 未显式 `base`
- **现状**：仅配置 `UnoCSS + sveltekit` 插件。
- **风险**：若 Cloudflare Pages 项目 URL 前缀、预览域或后续子路径部署发生变化，易出现静态资源 404/白屏。
- **建议修复**：在 `apps/web/vite.config.ts` 增加显式 `base`（默认 `'/'`），并在静态资源引用上统一遵循 `import.meta.env.BASE_URL`。

### 差异 C：环境变量占位不完整
- **现状**：
  - 根 `.env.example`：`API_URL`、`CF_PAGES`
  - `apps/web/.env.example`：`PUBLIC_API_URL`、`PUBLIC_USE_MOCK`
- **风险**：与 P1.2 目标中的 `KV_NAMESPACE`、`D1_DATABASE_ID` 等占位不一致，后续切 P1.3/P2.x 易发生“环境缺项”。
- **建议修复**：
  1. 在根 `.env.example` 增加：`KV_NAMESPACE=`、`D1_DATABASE_ID=`（如仅模板可为空）。
  2. 在部署文档中区分：
     - **Pages 可见变量（PUBLIC_ 前缀）**
     - **Workers/服务端密钥变量（不暴露到前端）**

### 差异 D：Pages 平台配置未在仓库内固化
- **现状**：仓库中无 P1.2 部署参数记录文件。
- **风险**：多人协作时 Cloudflare 控制台配置可能漂移（根目录/构建命令/输出目录/Preview 分支）。
- **建议修复**：新增部署运行手册（建议路径 `deploy/`），明确：
  - Root directory: `apps/web`
  - Build command: `pnpm --filter web build`
  - Build output directory: `.svelte-kit/cloudflare`
  - Preview branch: `dev`
  - 变量注入清单：`API_URL`、`KV_NAMESPACE`、`D1_DATABASE_ID`...
